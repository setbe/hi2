Deterministic Memory Model

In a::, memory is explicit. You define requirements first, then provide real storage, and execution stays inside those boundaries.

This chapter teaches the model in the same order you use it in code: requirement calculus, compile-time composition, runtime frame usage, and contract enforcement.

sum vs max

req_sum(a,b)

Use when components coexist in the same layout.

start_b = align_up(a.bytes, b.align)
result.bytes = start_b + b.bytes
result.align = max(a.align, b.align)

Line-by-line:
1) start_b = align_up(a.bytes, b.align) computes where b is allowed to start after a with proper alignment.
2) result.bytes = start_b + b.bytes gives total packed size after placing b.
3) result.align = max(a.align, b.align) keeps the strictest alignment required by the combined layout.
Plain example: if block A ends at byte 20 and block B requires align 16, B starts at byte 32, not 21.

req_max(a,b)

Use when phases are mutually exclusive and reuse one arena.

result.bytes = max(a.bytes, b.bytes)
result.align = max(a.align, b.align)

Line-by-line:
1) result.bytes = max(a.bytes, b.bytes) picks only the larger byte requirement because phases do not coexist.
2) result.align = max(a.align, b.align) keeps alignment safe for either phase when reusing one arena.
Plain example: if boot needs 96 KB and runtime needs 40 KB, reserve 96 KB total.

Rule: sum for coexistence, max for exclusive phases.

Worked Numeric Example

// A = { bytes: 20, align: 4  }
// B = { bytes: 48, align: 16 }
// C = { bytes: 12, align: 8  }

ab_start = align_up(20, 16) = 32
ab      = { bytes: 32 + 48 = 80, align: max(4,16) = 16 }

abc_start = align_up(80, 8) = 80
abc       = { bytes: 80 + 12 = 92, align: max(16,8) = 16 }

Interpretation: 92 bytes is the packed coexistence requirement for A+B+C. If these were exclusive phases instead, req_max would choose the largest bytes/alignment pair.

Compile-Time Composition by Types

struct font_phase {
    static constexpr a::memory_req requirement() noexcept { return {32 * 1024, 16}; }
};

struct image_phase {
    static constexpr a::memory_req requirement() noexcept { return {96 * 1024, 16}; }
};

using load_layout = a::layout<font_phase, image_phase>;
constexpr a::memory_req load_req = load_layout::requirement();

using boot_req = a::sum_req<font_phase, image_phase>;
using runtime_req = a::max_req<font_phase, image_phase>;

This example shows type-level memory contracts: every type exposes requirement(), and composition happens fully at compile time.
Plain example: font_phase says "32 KB", image_phase says "96 KB", and the compiler precomputes the total before run.

From Compile-Time Plan to Runtime Arena

using decode_layout = a::layout<font_phase, image_phase>;
constexpr a::memory_req decode_req = decode_layout::requirement();

a::u8 storage[decode_req.bytes] = {};
a::arena_view arena{storage, sizeof(storage)};

a::typed_scope<decode_layout> frame(arena);
if (!frame.active()) {
    return;
}

Pipeline: type-level requirement -> fixed storage -> arena binding -> typed scope activation.
Plain example: the compiler tells you exact minimum bytes, then runtime only checks that the provided arena satisfies that contract.

typed_scope Lifecycle Rules

  • active() tells if the scope is usable; init_status() stores start-up outcome only.
  • slot<T>() returns raw storage; no object lifetime is started.
  • construct<T>() starts lifetime with placement new and marks construction state.
  • destroy<T>() destroys only objects that were actually constructed.
  • destroy_all() is cleanup only; it does not close the scope.
  • release() closes the frame and makes the scope inert.
  • data() is an unsafe escape hatch that bypasses lifetime discipline.

Common Failure Modes

  • Using req_max where components are actually alive at the same time.
  • Assuming destroy_all() closes a typed scope (it does not).
  • Treating init_status() as a perpetual validity flag.
  • Bypassing typed lifecycle discipline via raw data() without strict local reasoning.

Context Strategy

A practical pattern is to keep two levels:

  • local sum_req/layout for simultaneously alive components;
  • global max_req for peak memory across exclusive contexts.
using boot = a::sum_req<font_phase, image_phase>;
using game = a::sum_req<renderer_state, ui_state>;
using app_peak = a::max_req<boot, game>;

This pattern separates local simultaneous layouts (sum_req) from global peak planning (max_req) for deterministic reuse.
Plain example: inside one phase you sum required blocks, across different phases you keep only the larger peak.
