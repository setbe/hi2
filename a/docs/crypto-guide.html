a::crypto Guide

a::crypto is intentionally minimal: no OS dependency, no hidden heap, and explicit caller-owned buffers from input to output.

Design Rules

  • Secrets stay in caller-managed memory; there are no hidden heap containers.
  • Data paths are built around a::byte_view/a::view-style interfaces.
  • Sensitive key material is wiped through crypto/detail/wipe.hpp.
  • Critical comparisons use constant-time helpers in crypto/detail/ct.hpp.
  • LE/BE access is centralized in crypto/detail/load_store.hpp on top of a::bit.

Recommended Usage Pattern

a::crypto::aead_chacha20_poly1305::key key{};
a::crypto::aead_chacha20_poly1305::nonce nonce{};
a::byte_view aad = ...;
a::byte_view plaintext = ...;
a::byte_view ciphertext = ...;
a::byte_view tag = ...;

const bool ok_seal = a::crypto::aead_chacha20_poly1305::seal(
    key, nonce, aad, plaintext, ciphertext, tag
);

const bool ok_open = a::crypto::aead_chacha20_poly1305::open(
    key, nonce, aad, ciphertext, tag, plaintext
);
if (!ok_open) {
    // authentication failed
}

The flow is: seal with key+nonce+AAD to produce ciphertext/tag, then open with the same context; failure means authentication did not pass.
Plain example: if one packet byte changes in transit, open() fails and plaintext is rejected.

Integration with a::two_pass

In larger pipelines (batch verification, handshake phases, asset protection), cryptographic steps can be modeled as plan/build stages, with scratch memory managed by a::typed_scope.

Practice: keep key lifetime as short as possible and wipe immediately after use.

Security Notes

  • Never reuse nonce values with ChaCha20-Poly1305.
  • Validate MAC/tag before treating plaintext as trusted.
  • Avoid unnecessary copies of sensitive buffers; in-place paths are usually safer.
  • Use unsafe escape hatches only when the surrounding contract is tightly controlled.
