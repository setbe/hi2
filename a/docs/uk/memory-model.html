Deterministic Memory Model

У a:: пам'ять не "магічна": спершу формулюємо контракт, далі даємо реальний буфер і працюємо строго в його межах. Такий підхід робить поведінку передбачуваною навіть у найжорсткіших bare-metal сценаріях.

sum vs max

req_sum(a,b)

Коли компоненти мають жити одночасно в одному layout.

start_b = align_up(a.bytes, b.align)
result.bytes = start_b + b.bytes
result.align = max(a.align, b.align)

Ця формула вирівнює другий компонент, додає його байти та зберігає найсуворіше вирівнювання для всього packed layout.
На пальцях: якщо перший блок закінчився на байті 20, а другому треба align 16, другий стартує з 32, а не з 21.

req_max(a,b)

Коли фази взаємовиключні й arena перевикористовується.

result.bytes = max(a.bytes, b.bytes)
result.align = max(a.align, b.align)

Ця формула не пакує компоненти разом; вона рахує піковий reusable region для взаємовиключних фаз.
На пальцях: якщо boot просить 96 КБ, а runtime 40 КБ, загалом треба лише 96 КБ.

Правило: sum для співіснування, max для взаємовиключних фаз.

Compile-Time Composition by Types

struct font_phase {
    static constexpr a::memory_req requirement() noexcept { return {32 * 1024, 16}; }
};

struct image_phase {
    static constexpr a::memory_req requirement() noexcept { return {96 * 1024, 16}; }
};

using load_layout = a::layout<font_phase, image_phase>;
constexpr a::memory_req load_req = load_layout::requirement();

using boot_req = a::sum_req<font_phase, image_phase>;
using runtime_req = a::max_req<font_phase, image_phase>;

У прикладі контракти пам'яті задаються на рівні типів: кожен тип має requirement(), а композиція виконується на compile-time.
На пальцях: тип font_phase каже "32 КБ", image_phase каже "96 КБ", а компілятор одразу знає суму.

typed_scope Lifecycle Rules

  • active() визначає, чи scope usable; init_status() зберігає лише результат старту.
  • slot<T>() дає raw storage slot, без старту lifetime.
  • construct<T>() запускає lifetime через placement new і фіксує constructed-стан.
  • destroy<T>() руйнує лише реально сконструйований об'єкт.
  • destroy_all() чистить об'єкти, але не закриває scope.
  • release() закриває frame, звільняє lifetime і робить scope inert.
  • data() — свідомий unsafe escape hatch, який обходить safety-дисципліну.

Context Strategy

У практичних системах зазвичай працює двошаровий підхід:

  • sum_req/layout для локальних одночасно живих компонентів;
  • max_req для глобального peak між взаємовиключними контекстами.
using boot = a::sum_req<font_phase, image_phase>;
using game = a::sum_req<renderer_state, ui_state>;
using app_peak = a::max_req<boot, game>;

Цей патерн ділить локальні одночасні layout-и (sum_req) і глобальне пікове планування (max_req) для детермінованого reuse.
На пальцях: всередині однієї фази складаєш блоки, між різними фазами береш просто більший пік.
