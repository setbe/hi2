a::two_pass Protocol Guide

a::two_pass — це дисципліна планування, а не просто ще один API. Спочатку рахуємо і валідуємо вимоги, потім будуємо результат у заданій arena, без випадкових алокацій і розмитих error-сигналів.

How to implement module-level two-pass

struct image_input {
    a::byte_view encoded;
    a::u32 max_w;
    a::u32 max_h;
};

struct image_plan {
    a::u32 width;
    a::u32 height;
    a::memory_req scratch;
};

struct image_output {
    a::byte_view pixels;
};

namespace a::two_pass {
template<>
struct traits<image_plan, image_input> {
    static plan_result<image_plan> plan(const image_input& in) noexcept;
    static build_result<image_output> build(const image_plan& p,
                                             a::arena_view arena,
                                             a::byte_view out_pixels) noexcept;
};
}

Це каркас протоколу: plan() валідовує вхід і форму пам'яті, а build() будує результат лише за затвердженим планом.
На пальцях: спочатку перевірив заголовок PNG і порахував scratch, потім уже декодуєш.

Status Discipline

  • Не зводь помилки до bool: повертай a::status.
  • Брак пам'яті сигналізуй через status::insufficient_memory.
  • Порушення контракту: invalid_input, mismatch, overflow.
  • Помилки декодера/бекенда: unsupported або backend_failure.

Composition Helpers

Static components

using req = a::sum_req<
    a::two_pass::static_req<4096, 16>,
    a::two_pass::static_req<8192, 16>
>;

Тут збирається повністю статичний bundle вимог, де розмір і вирівнювання відомі ще до запуску.
На пальцях: поєднав 4 КБ і 8 КБ блоки і ще до запуску знаєш, який мінімум пам'яті потрібен.

Context selection

using boot = a::sum_req<font_plan, atlas_build>;
using run  = a::sum_req<frame_state, ui_state>;
using peak = a::max_req<boot, run>;

Цей приклад поєднує два контексти й обчислює один піковий memory-контракт для взаємовиключного виконання.
На пальцях: меню просить 24 КБ, гра 80 КБ, значить одна арена на 80 КБ достатня.

Contract Checklist

  • Plan-фаза не модифікує user buffers.
  • Build-фаза не виходить за межі memory, зафіксовані у плані.
  • Тимчасові буфери живуть у a::arena_view і мають контрольований lifetime.
  • Секрети в crypto-шляхах очищаються через wipe helper.
  • Публічний API не містить platform/OS типів.
