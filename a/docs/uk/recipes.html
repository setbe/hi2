Practical Recipes

Це робочий зошит, а не просто збірка фрагментів. Кожен рецепт містить і код, і пояснення контракту.

Перед Стартом

  • Дивись на рецепт як на контрактний патерн, а не лише на синтаксис.
  • Тримай статуси явними; не стискай помилки до bool.
  • sum_req — лише для співіснування, max_req — лише для взаємовиключних фаз.

1. Compile-time peak memory across contexts

struct boot_a { static constexpr a::memory_req requirement() noexcept { return {24 * 1024, 16}; } };
struct boot_b { static constexpr a::memory_req requirement() noexcept { return {40 * 1024, 16}; } };
struct run_a  { static constexpr a::memory_req requirement() noexcept { return {80 * 1024, 16}; } };
struct run_b  { static constexpr a::memory_req requirement() noexcept { return {32 * 1024, 16}; } };

using boot_ctx = a::sum_req<boot_a, boot_b>;
using run_ctx  = a::sum_req<run_a, run_b>;
using peak_ctx = a::max_req<boot_ctx, run_ctx>;

constexpr a::memory_req app_req = peak_ctx::requirement();

Тут рахується детермінована верхня межа пам'яті: контексти пакуються через sum_req, а глобальний максимум береться через max_req.
На пальцях: boot може просити 64 КБ, gameplay 96 КБ, отже виділяєш 96 КБ.

2. Typed slots with explicit construction

struct parser_state {
    static constexpr a::memory_req requirement() noexcept { return {sizeof(parser_state), alignof(parser_state)}; }
    parser_state() noexcept = default;
    ~parser_state() noexcept = default;
};

using parser_layout = a::layout<parser_state>;

void run_parser(a::arena_view arena) {
    a::typed_scope<parser_layout> frame(arena);
    if (!frame.active()) return;

    parser_state* st = frame.construct<parser_state>();
    if (!st) return;

    // work

    frame.destroy<parser_state>();
    frame.release();
}

Цей рецепт поєднує явне очищення і RAII: destroy<T>() одразу руйнує конкретний об'єкт, а release() закриває frame і робить scope inert.
На пальцях: розібрав один файл, викликав destroy<parser_state>() щоб завершити саме цей об'єкт зараз, потім release() щоб закрити весь scope; якщо забув, у ~typed_scope() все одно спрацює destroy_all() як safety fallback.

3. Reusable scratch with arena scope

void decode_batch(a::arena_view arena) {
    {
        a::scope phase(arena);
        auto r = arena.alloc(48 * 1024, 16);
        if (!a::is_ok(r.code)) return;
        // decode image set A
    } // rewind to phase mark

    {
        a::scope phase(arena);
        auto r = arena.alloc(48 * 1024, 16);
        if (!a::is_ok(r.code)) return;
        // decode image set B using same bytes
    }
}

Кожен внутрішній a::scope автоматично повертає arena offset, тому scratch-пам'ять безпечно перевикористовується між фазами.
На пальцях: декодував іконку A, scope закрився, далі в ті самі 48 КБ декодуєш іконку B.

4. Two-pass module skeleton

namespace my {
struct plan { a::u32 count; };
struct input { a::u32 max_count; };
struct output { a::u32 written; };
}

namespace a::two_pass {
template<>
struct traits<my::plan, my::input> {
    static plan_result<my::plan> plan(const my::input& in) noexcept {
        if (in.max_count == 0) return {status::invalid_input, {}, {}};
        my::plan p{in.max_count};
        return {status::ok, p, {in.max_count * sizeof(a::u32), alignof(a::u32)}};
    }

    static build_result<my::output> build(const my::plan& p, a::arena_view arena) noexcept {
        auto r = arena.alloc(p.count * sizeof(a::u32), alignof(a::u32));
        if (!is_ok(r.code)) return {r.code, {0}};
        return {status::ok, {p.count}};
    }
};
}

Цей шаблон демонструє повний two-pass контракт: спочатку валідація у plan(), потім алокація і побудова у build().
На пальцях: спочатку порахував рядки CSV, потім виділив точний буфер і записав результат.

5. Minimal crypto flow

a::crypto::sha256 h;
h.update(data_a);
h.update(data_b);
a::crypto::sha256::digest d = h.final();

// optionally reuse object
h.reset();

Це мінімальний шаблон потокового хешування: оновлюємо чанками, завершуємо digest, за потреби робимо reset для повторного використання.
На пальцях: читаєш файл шматками по 4 КБ, кожен шматок у update(), в кінці один final().

Як Перетворити Рецепт на Продакшен Модуль

  1. Винеси вимоги в окремі named spec-типи.
  2. Зафіксуй стабільну мапу помилок у a::status.
  3. Додай compile-time перевірки для ключових layout припущень.
  4. Додай runtime тести для normal path і memory-pressure path.
