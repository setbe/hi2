a::crypto Guide

Криптографія в a::crypto спроєктована в стилі "нічого зайвого": жодної залежності від ОС, жодного прихованого heap, лише чіткий API над caller-owned буферами.

Ціль цього розділу — не просто перелік алгоритмів, а практична дисципліна використання крипто в детермінованій low-level системі.

Design Rules

  • Секрети живуть у зовнішніх буферах; бібліотека не тримає приховані heap-контейнери.
  • Поток даних іде через a::byte_view/a::view-подібні інтерфейси.
  • Ключі й чутливий state очищаються через crypto/detail/wipe.hpp.
  • Порівняння критичних буферів робиться константно-часово через crypto/detail/ct.hpp.
  • LE/BE доступ централізований у crypto/detail/load_store.hpp поверх a::bit.

Recommended Usage Pattern

a::crypto::aead_chacha20_poly1305::key key{};
a::crypto::aead_chacha20_poly1305::nonce nonce{};
a::byte_view aad = ...;
a::byte_view plaintext = ...;
a::byte_view ciphertext = ...;
a::byte_view tag = ...;

const bool ok_seal = a::crypto::aead_chacha20_poly1305::seal(
    key, nonce, aad, plaintext, ciphertext, tag
);

const bool ok_open = a::crypto::aead_chacha20_poly1305::open(
    key, nonce, aad, ciphertext, tag, plaintext
);
if (!ok_open) {
    // authentication failed
}

Послідовність така: seal з key+nonce+AAD дає ciphertext/tag, open перевіряє їх; помилка означає провал автентифікації.
На пальцях: якщо в пакеті змінився хоч один байт, open() поверне помилку і plaintext не приймається.

Шаблон Streaming-Hash

a::crypto::sha256 h;

while (read_chunk(chunk)) {
    h.update(chunk);
}

a::crypto::sha256::digest d = h.final();
h.reset();

Патерн: update чанками, final один раз, reset лише для нового логічного повідомлення.

Integration with a::two_pass

У складних пайплайнах (batch verification, handshake phases, asset protection) криптокроки зручно формалізувати як plan/build етапи, а тимчасову пам'ять керувати через a::typed_scope.

Практика: тримай ключі в найкоротшому можливому scope і зануляй одразу після використання.

Коли Потрібні Fixed-Size Views

  • Ключі: розмір зафіксований алгоритмом.
  • Nonce/IV: розмір зафіксований режимом.
  • Tag/MAC: довжина визначена специфікацією алгоритму.

Це переносить size-валидацію з runtime-розгалужень на межу типу.

Security Notes

  • Не повторюй nonce для ChaCha20-Poly1305.
  • Перевіряй MAC/tag до того, як вважати plaintext довіреним.
  • Уникай зайвих копій секретних буферів; in-place обробка зазвичай краща.
  • Unsafe escape hatch використовуйте лише там, де контракт реально контрольований.
  • Явно фіксуй ownership ключів: хто створює, хто використовує, хто очищає.
