Required C++ Knowledge for Starting with Bare-Metal

Цей розділ написаний як технічний вступ-підручник, а не як сухий список. Ми починаємо з відомих C++ понять і поступово занурюємось у глибину, на якій працює a::.

Словник Перед Стартом (Прочитай Першим)

Ці терміни зустрічаються у всьому a/. Якщо зафіксувати їх зараз, далі матеріал читається значно легше.

Айсберг Рівень 1: Компіляція, Лінкування, Виконання

У цій архітектурі compile-time - це частина коректності, а не просто оптимізація. Ми свідомо переносимо багато рішень у compile-time, щоб поведінка була передбачуваною. 

constexpr a::u32 kib(a::u32 n) noexcept { return n * 1024u; }

constexpr a::u32 boot_pool = kib(64); // compile-time обчислення

a::u32 user_pages = 3;
a::u32 runtime_pool = kib(user_pages); // runtime обчислення

Ключ: constexpr означає "може бути compile-time", а не "завжди compile-time". Та сама функція може виконатись і на етапі компіляції, і під час виконання.
На пальцях: відомо зараз -> рахує компілятор, стане відомо потім -> рахує CPU.

Айсберг Рівень 2: Lifetime та Ownership (Чому RAII - Фундамент)

a:: тримається на явних memory-контрактах і детермінованому cleanup. RAII робить це стабільним навіть при ранніх return і складних гілках. 

void parse_phase(a::arena_view& arena) {
    a::scope frame(arena); // отримали mark memory-frame
    // тимчасові алокації і тимчасові об'єкти
} // деструктор робить rewind, ownership фази завершується

Технічно: lifetime і ownership зв'язані: власник frame відповідає за його завершення.
На пальцях: відкрив робочий стіл, зробив тимчасову роботу, закрив стіл, прибрав усе разом.

Айсберг Рівень 3: Вказівники, Runtime-Розмір і Compile-Time-Розмір

Вказівник дає гнучкість, але сам по собі не несе достатньо контракту: він містить адресу, але не довжину і не семантику буфера. 

a::u8 raw[256]{};
a::usize n = 128;
a::view<a::u8> dyn{raw, n}; // view з runtime-розміром

Саме тому "просто використати std::vector" не вирішує детерміновані протокольні вимоги: розмір vector - це runtime-стан, а багато крипто/бінарних контрактів мають фіксовані розміри зі специфікації.

Айсберг Рівень 4: Що Таке View Насправді (view<T> проти view_n<T,N>)

View - це не-owning вікно у вже існуючу пам'ять. У a/ це ключове розділення: dynamic view для runtime-даних і fixed view для compile-time контрактів. 

a::u8 key[32]{};
a::u8 nonce[12]{};
a::u8 tag[16]{};

a::byte_view_n<32> key_v = a::as_view(key);
a::byte_view_n<12> nonce_v = a::as_view(nonce);
a::byte_view_mut_n<16> tag_v = a::as_view_mut(tag);

Чому це size-safe: розмір зашитий у тип. Неправильний буфер не "майже підійде", а буде відсічений на межі типу.
Реальний приклад: a::crypto::x25519 і a::crypto::aead_chacha20_poly1305 використовують fixed-size view для ключів/nonce/tag.

Перед padding/alignment треба зафіксувати мову типів. У цьому проєкті імена a:: є явними alias поверх базових інтегральних типів C++.

a::u16unsigned shortКомпактні лічильники/поля.
a::u32unsigned intРозміри, лічильники, протокольні слова.
a::u64unsigned long longВеликі лічильники і wide-арифметика.
a::usizedecltype(sizeof(0))Розміри об'єктів і кількість байтів у memory API.

Навіщо alias-и: вони роблять контракти API читабельними.
На пальцях: a::u8 одразу каже "байтова семантика", а не "просто якесь число".

Айсберг Рівень 5: Padding та Alignment (Практично і Глибоко)

Alignment - це закон адресації, padding - наслідок цього закону в layout. Якщо ігнорувати хоч одне з двох, memory-математика рано чи пізно стає хибною.

У фрагменті нижче імена беруться з a/types.hpp: a::u8 це unsigned char, а a::u32 це unsigned int. 

struct packet_header {
    a::u8  type;   // offset 0
    a::u32 size;   // часто offset 4, а не 1
};

// Типово на поширених ABI:
// alignof(packet_header) == 4
// sizeof(packet_header)  == 8 (через padding)

Що важливо запам'ятати:

  • Offset поля не дорівнює "просто сума попередніх полів".
  • Stride масиву структур = sizeof(T), включно з tail padding.
  • Ручний packed-рахунок без alignment правил ламає контракт layout.
// Логіка, яку використовує req_sum(a, b):
start_b = align_up(a.bytes, b.align)
result.bytes = start_b + b.bytes
result.align = max(a.align, b.align)

На пальцях: якщо A закінчився на байті 20, а B вимагає align 16, B стартує з 32, не з 21. Пропущені байти - це структурна вимога, а не випадковий "мусор".

Айсберг Рівень 6: Memory-Frame та Rewind

Memory-frame - це scoped-обіцянка: "все після цієї позначки тимчасове". Rewind виконує цю обіцянку однією операцією. 

a::u8 storage[64 * 1024]{};
a::arena_view arena{storage, sizeof(storage)};

{
    a::scope frame(arena); // зберегли mark
    // alloc A, alloc B, alloc C
} // rewind: A/B/C прибираються разом

Чому це масштабується: без списків free, без прихованих графів ownership. Один mark, один rewind, детермінована вартість.

Айсберг Рівень 7: Контракти (require, assume, assume_verified)

Контракти - це мова істини в цьому фреймворку. Вони розділяють обов'язок caller-а, гарантії модуля і припущення для оптимізатора.

ПримітивВикористовувати Саме ДляНе Використовувати Для
require(cond)Передумова на межі APIСтилістичні або опційні підказки
ensure(cond)Постумова після операціїПеревірка користувацького вводу на вході
invariant(cond)Внутрішня консистентність об'єкта/модуляОбробка зовнішньої помилки користувача
assume(cond)Перевірене припущення з optional optimizer-supportЗаміна семантики require
assume_verified(cond)Умова вже доведена і стабільна в цьому шляхуНедоведена або змінна умова

Trusted path: шлях, у якому умова вже встановлена і не може змінитися до місця використання. Приклад: після require(is_pow2(align)), у гілці без зміни align, коректно давати assume_verified(is_pow2(align)).

Айсберг Рівень 8: Freestanding vs Bare-Metal, і Чому a:: Не Алокує RAM

Freestanding означає мінімальні припущення про runtime/бібліотеку. Bare-metal означає роботу близько до апаратного середовища, часто без повної ОС.

Рівень a принципово чистий: він не може припускати OS allocator, virtual memory, filesystem, thread scheduler. Тому отримання RAM із середовища - не його задача.

Контракт шарів: a задає детерміновані примітиви й працює з caller-provided пам'яттю. b може дати OS/platform джерела пам'яті. c будує high-level фреймворки зверху.

На пальцях: a - це геометрія двигуна, а не заправка пальним.
