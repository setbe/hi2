API Атлас

Якщо Module Map показує архітектуру, то Atlas показує контрактну карту: які типи існують, що саме вони гарантують і як їх правильно комбінувати.

typed_scope Invariants

Usability

active() / operator bool() визначають, чи scope робочий. init_status() зберігає лише статус ініціалізації.

Cleanup vs Close

destroy_all() чистить об'єкти, але не закриває scope. release() закриває scope й робить його inert.

Type Safety

slot/get/construct/destroy статично гарантують, що T належить Layout.

Nothrow Policy

Для побудови в typed_scope компоненти мають бути nothrow-constructible і nothrow-destructible.

Основний інваріант: якщо active()==false, scope не usable; inert-стан означає порожній constructed-state.

a/bounded.hppbounded, pow2_bounded, static_boundedРозміри/count/alignment як валідовані типи
  • require: передумови на межі API/caller.
  • require_that<Tag>: runtime-перевірка з випуском proof-token.
  • assume_verified(proven<Tag>&&): одноразове споживання proof-token у trusted path.
  • Trusted path у bounded-типах є internal-only конструюванням, а не шляхом обходу валідації user input.

Як Практично Користуватися Атласом

  • Під час імплементації: спочатку звір намір API з таблицями тут, потім пиши код.
  • Під час рев'ю: перевір, що нові символи не ламають уже зафіксовані інваріанти.
  • Під час дебагу: спочатку класифікуй порушення (precondition/invariant/lifetime), потім шукай точку поломки.
