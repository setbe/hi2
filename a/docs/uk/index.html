Документація a/ і публічного API a::

Ключові слова: dependency law, a→b→c→d, freestanding, bare-metal, determinism, capabilities, contracts, two-pass, arena, typed_scope.

Ця документація задумана як навчальний текст, а не короткий конспект. Вона пояснює не тільки “які є хедери”, а й “чому ця архітектура саме така” і “як її безпечно розвивати”.

Модель шарів проста: a — сувора база, b — capability middleware, c — portable high-level framework layer, d — application model. На цій секції фокус саме на a, бо він задає правила для всіх рівнів вище.

Архітектура в Одній Картинці

a:: Суворий Core

Типи, view, memory algebra, контракти, криптопримітиви. Без OS includes, без hidden heap.

Deterministic Freestanding-first

b:: Capability Middleware

Шар взаємодії із середовищем (threads/time/net/windowing), який може залежати від a.

Platform-aware

c:: Portable Frameworks

Вищі portable абстракції, які спираються на можливості b і правила a.

Capability-driven

d:: App Model

Найвищий рівень (декларативні/reactive моделі), який будується поверх нижчих гарантій.

Highest abstraction

Dependency law: a не залежить ні від чого, b залежить тільки від a, c — від a+b, d — від a+b+c.

Головні Обіцянки a::

  • Пам'ять явна: storage передає caller, бібліотека не “дістає пам'ять сама”.
  • Контракти явні: preconditions, invariants і impossible states мають один формальний механізм.
  • Композиція явна: req_sum і req_max означають різні memory-семантики.
  • Lifetime явний: frame/scope і typed construction/destruction завжди видно в коді.
  • Crypto явне: caller-owned buffers і size-safe межі.

Quick Start (З Поясненням)

#include "a/a.hpp"

struct font_scratch {
    static constexpr a::memory_req requirement() noexcept { return {32 * 1024, 16}; }
};

struct image_scratch {
    static constexpr a::memory_req requirement() noexcept { return {64 * 1024, 16}; }
};

using decode_layout = a::layout<font_scratch, image_scratch>;

void run(a::arena_view arena) {
    a::typed_scope<decode_layout> frame(arena);
    if (!frame.active()) {
        return;
    }

    auto* font_mem = frame.slot<font_scratch>();
    auto* image_mem = frame.slot<image_scratch>();
    (void)font_mem;
    (void)image_mem;

    frame.release();
}

Що відбувається по кроках:
1) Типи-компоненти задають memory-контракт.
2) layout<...> на compile-time рахує offsets і загальний розмір.
3) typed_scope прив'язує цей layout до реальної arena.
4) slot<T>() повертає typed storage для кожного блоку.
5) release() закриває frame і робить scope inert.
На пальцях: один буфер на старті, два scratch-блоки всередині, завершення фази без hidden allocations.

Що Це Точно Не Є

  • Не прихований runtime, який сам “якось керує пам'яттю”.
  • Не заміна platform-шару b, який дає доступ до реальних OS capabilities.
  • Не набір стилістичних порад, які можна ігнорувати без наслідків.
