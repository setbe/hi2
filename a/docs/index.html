Documentation for a/ and the public a:: API

Keywords: dependency law, a?b?c?d, freestanding, bare-metal, determinism, capabilities, contracts, two-pass, arena, typed_scope.

This documentation is written as a learning guide, not a short cheat sheet. The goal is to make the architecture understandable to a newcomer and still rigorous enough for contributors who maintain low-level code.

The central model is simple and strict: a is the deterministic foundation, b is capability middleware, c is portable high-level framework space, and d is the application model layer. This website focuses on a, because every higher layer inherits its rules.

Who This Is For

Students Learning Systems C++

You know basic C++, but want a practical, deterministic model without std::-centric habits. Start with C++ Knowledge, then Memory Model.

Engine and Firmware Developers

You need strict ownership, no hidden allocation, and explicit contracts that scale under pressure. Start with Modules and Two Pass.

Contributors and Reviewers

You need precise API invariants and line-level reference. Read API Atlas, then verify exact declarations in Source.

Architecture in One Page

a:: Strict Core

Core types, views, memory algebra, contracts, and crypto primitives. No OS includes, no hidden heap, no scheduler assumptions.

Deterministic Freestanding-first

b:: Capability Middleware

Environment-facing layer (threads, time, sockets, windowing, etc.) exposed through capability contracts. b may depend on a, never the other way around.

Platform-aware

c:: Portable Frameworks

High-level abstractions that target portability by consuming capabilities from b and deterministic foundations from a.

Capability-driven

d:: App Model

The application-facing layer for declarative/reactive programming styles. It is intentionally built on top of the lower guarantees, not as a replacement for them.

Highest abstraction

Dependency law: a depends on nothing, b may depend on a, c may depend on a+b, d may depend on a+b+c.

Core Promises of a::

  • Memory is explicit: you pass storage in, the API does not silently obtain storage for you.
  • Contracts are explicit: preconditions, invariants, and impossible states are modeled by one system.
  • Composition is explicit: req_sum and req_max express different semantics.
  • Lifetime is explicit: scoped rewind and typed construction/destruction are visible in code.
  • Crypto is explicit: caller-owned buffers and deterministic, size-safe boundaries.

Quick Start (Step by Step)

#include "a/a.hpp"

struct font_scratch {
    static constexpr a::memory_req requirement() noexcept { return {32 * 1024, 16}; }
};

struct image_scratch {
    static constexpr a::memory_req requirement() noexcept { return {64 * 1024, 16}; }
};

using decode_layout = a::layout<font_scratch, image_scratch>;

void run(a::arena_view arena) {
    a::typed_scope<decode_layout> frame(arena);
    if (!frame.active()) {
        return;
    }

    auto* font_mem = frame.slot<font_scratch>();
    auto* image_mem = frame.slot<image_scratch>();
    (void)font_mem;
    (void)image_mem;

    frame.release();
}

Line-by-line intent:
1) Each component declares compile-time memory requirement.
2) layout<...> computes offsets and total requirement at compile-time.
3) typed_scope binds that layout to caller-provided arena memory.
4) slot<T>() returns typed storage for each component.
5) release() closes the frame and makes the scope inert.
Plain example: one fixed boot-time buffer is reused for temporary font and image work, with no hidden allocation.
