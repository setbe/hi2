Module Map

This chapter explains the role of every public header in a/, the intended dependency direction, and the mental model you should keep while extending the codebase.

How to Read This Map

Think in two axes:

  • Layer axis: what must stay pure (a) vs what can depend on environment.
  • Domain axis: memory, contracts, views, math, crypto, and composition.

Rule of thumb: if a header requires OS types or hidden allocation, it does not belong in a.

Dependency Order Inside a/

config
  -> types
  -> meta / utility
  -> view / utf8 / bit / math / charconv_base
  -> memory_requirements / arena
  -> two_pass
  -> crypto

This order keeps low-level primitives independent and reusable.
Plain example: crypto/detail/load_store.hpp depends on a::bit so byte-order logic is implemented once.

Internal Headers (detail/ Surface)

a/detail/arch.hpp

Arch tags and arch_default selection for internal fast-path policy. This is where architecture-specific specialization belongs, not in public names.

a/crypto/detail/load_store.hpp

Canonical LE/BE load-store helpers for crypto modules, built on a::bit. This avoids drift in byte-order behavior across algorithms.

a/crypto/detail/ct.hpp

Constant-time compare helpers: ct_equal_n and ct_equal. These exist to prevent accidental timing leaks from regular early-exit comparisons.

a/crypto/detail/wipe.hpp

Best-effort secret wipe via volatile byte loop. Use this for keys and intermediate secrets that should not persist after phase completion.

Common Placement Mistakes

  • Putting thread/scheduler behavior in a instead of b.
  • Adding APIs that silently allocate memory behind callers.
  • Mixing protocol logic and OS wrapper logic in one header.
  • Leaking platform-native handles into portable public interfaces.
