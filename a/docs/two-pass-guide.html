a::two_pass Protocol Guide

a::two_pass is a planning discipline, not just another API. First you validate and size memory, then you build into a known arena region.

Core Types

TypeRole
a::two_pass::plan_result<Plan>status code + Plan plan + memory_req memory
a::two_pass::build_result<Output>status code + Output value
a::two_pass::build_result<void>Build-only status without payload
a::two_pass::traits<Plan,Input>Customization point for concrete modules
a::two_pass::static_req<Bytes,Align>Compile-time leaf requirement
a::two_pass::scope<Ts...>Wrapper over a::typed_scope<a::layout<Ts...>>

How to implement a module-level two-pass flow

struct image_input {
    a::byte_view encoded;
    a::u32 max_w;
    a::u32 max_h;
};

struct image_plan {
    a::u32 width;
    a::u32 height;
    a::memory_req scratch;
};

struct image_output {
    a::byte_view pixels;
};

namespace a::two_pass {
template<>
struct traits<image_plan, image_input> {
    static plan_result<image_plan> plan(const image_input& in) noexcept;
    static build_result<image_output> build(const image_plan& p,
                                             a::arena_view arena,
                                             a::byte_view out_pixels) noexcept;
};
}

This is the protocol skeleton: plan() validates input and memory shape, while build() consumes the approved plan and arena.
Plain example: plan checks image header and required scratch size, build decodes only if that plan succeeded.

Status Discipline

  • Do not collapse outcomes into bool; return a::status.
  • Use status::insufficient_memory for memory pressure.
  • Use invalid_input, mismatch, overflow for contract violations.
  • Use unsupported or backend_failure for backend/decode issues.

Composition Helpers

Static components

using req = a::sum_req<
    a::two_pass::static_req<4096, 16>,
    a::two_pass::static_req<8192, 16>
>;

This creates a fully static requirement bundle where size and alignment are known at compile time.
Plain example: combining 4 KB and 8 KB static pieces lets you know exact minimum memory before flashing firmware.

Context selection

using boot = a::sum_req<font_plan, atlas_build>;
using run  = a::sum_req<frame_state, ui_state>;
using peak = a::max_req<boot, run>;

This combines two context requirements and computes one peak memory contract for exclusive execution phases.
Plain example: menu context needs 24 KB, gameplay needs 80 KB, so one 80 KB arena is enough.

Contract Checklist

  • Plan phase does not mutate user buffers.
  • Build phase stays inside memory bounds defined by plan.
  • Temporary storage lives in a::arena_view.
  • Sensitive data is wiped in crypto paths.
  • Public API remains free of platform/OS types.
