Required C++ Knowledge for Starting with Bare-Metal

This chapter is written as a technical onboarding book, not as a checklist. We start from familiar C++ ground and descend into the deeper ideas that power a::.

Preflight Vocabulary (Read Before Everything Else)

These words appear across a/. Read them first, so later sections feel natural.

Iceberg Level 1: Compile, Link, Execute

In this architecture, compile-time is not a side topic: it is part of correctness. We deliberately move many decisions to compile-time because deterministic systems benefit from early guarantees. 

constexpr a::u32 kib(a::u32 n) noexcept { return n * 1024u; }

constexpr a::u32 boot_pool = kib(64); // compile-time evaluation

a::u32 user_pages = 3;
a::u32 runtime_pool = kib(user_pages); // runtime evaluation

Key point: constexpr means "can be compile-time", not "must be compile-time". The same function may execute in either stage.
On fingers: known now -> compiler computes it. Known later -> CPU computes it.

Iceberg Level 2: Lifetime and Ownership (Why RAII Is a Foundation)

a:: uses explicit memory contracts and deterministic cleanup. RAII is the mechanism that makes this reliable under early-return, errors, and refactors. 

void parse_phase(a::arena_view& arena) {
    a::scope frame(arena); // acquires a memory frame mark
    // temporary allocations and objects
} // destructor rewinds: frame ownership ends here

Technical view: lifetime and ownership are coupled: whoever owns the frame ends its lifetime.
On fingers: open a workbench, do temporary work, close the bench, reset the table.

Iceberg Level 3: Pointers, Runtime Size, and Compile-Time Size

Pointer-based APIs are flexible but weakly constrained. A pointer carries address, not element count, and never the semantic contract of the data. 

a::u8 raw[256]{};
a::usize n = 128;
a::view<a::u8> dyn{raw, n}; // runtime-sized view

This is one reason "just use std::vector" does not solve deterministic protocol shape: vector size is runtime state, while many cryptographic and binary contracts are fixed by specification.

Iceberg Level 4: What a View Really Is (view<T> vs view_n<T,N>)

A view is a non-owning window into already existing storage. In a/, this distinction is central: runtime-sized view for dynamic data, compile-time-sized view for fixed contracts. 

a::u8 key[32]{};
a::u8 nonce[12]{};
a::u8 tag[16]{};

a::byte_view_n<32> key_v = a::as_view(key);
a::byte_view_n<12> nonce_v = a::as_view(nonce);
a::byte_view_mut_n<16> tag_v = a::as_view_mut(tag);

Why this is size-safe: the size is encoded in the type itself. A wrong-sized buffer does not quietly "sort of work" - it fails at the type boundary.
Real usage: a::crypto::x25519 and a::crypto::aead_chacha20_poly1305 use fixed-size views for keys/nonces/tags.

Iceberg Level 5: Padding and Alignment (Deep Practical View)

Alignment is an address law; padding is a layout consequence. Ignore either one, and memory math eventually lies to you.

In the snippet below, type names are from a::types.hpp: a::u8 is unsigned char, and a::u32 is unsigned int. 

struct packet_header {
    a::u8  type;   // offset 0
    a::u32 size;   // often offset 4, not 1
};

// Typical result on common ABIs:
// alignof(packet_header) == 4
// sizeof(packet_header)  == 8 (includes padding)

Why this matters in deterministic planners:

  • Field offsets are not "sum of previous field sizes".
  • Array stride is sizeof(T), including tail padding.
  • Packed byte math without alignment rules eventually corrupts layout assumptions.
// Concept used by req_sum(a, b):
start_b = align_up(a.bytes, b.align)
result.bytes = start_b + b.bytes
result.align = max(a.align, b.align)

On fingers: if block A ends at 20 and block B needs alignment 16, B starts at 32, not 21. The skipped bytes are intentional structure, not waste.

Iceberg Level 6: Memory Frame and Rewind

A memory frame is a scoped promise: "everything allocated after this mark is temporary". Rewind enforces that promise in one operation. 

a::u8 storage[64 * 1024]{};
a::arena_view arena{storage, sizeof(storage)};

{
    a::scope frame(arena); // save mark
    // alloc A, alloc B, alloc C
} // rewind: A/B/C are collectively dropped

Why it scales: no per-object free list, no hidden ownership graph. One mark, one rewind, deterministic cost.

Iceberg Level 7: Contracts (require, assume, assume_verified)

Contracts are the language of truth in this framework. They separate caller obligations, internal guarantees, and optimization-grade assumptions.

PrimitiveUse Exactly ForAvoid
require(cond)Public/API preconditionOptional hints or style checks
ensure(cond)Postcondition after operationInput validation at boundary
invariant(cond)Object/module internal consistencyExternal user error handling
assume(cond)Checked assumption with optional optimizer supportReplacing require semantics
assume_verified(cond)Condition already proven and stable in the current pathUnproven or mutable condition

Trusted path: a path where the condition is already established and cannot change before use. Example: after require(is_pow2(align)), inside a tight branch that does not modify align, using assume_verified(is_pow2(align)) is valid.

Iceberg Level 8: Freestanding vs Bare-Metal, and Why a:: Does Not Allocate RAM

Freestanding means minimal language/runtime assumptions. Bare-metal means operating close to hardware, often without full OS services.

Layer a is intentionally pure: it cannot assume OS allocators, virtual memory, filesystem, or scheduler-backed services. Therefore, RAM acquisition is not a's job.

Layer contract: a defines deterministic primitives and consumes caller-provided storage. b may provide OS/platform memory sources. c builds higher-level frameworks on top.

On fingers: a is the engine geometry, not the gas station.
